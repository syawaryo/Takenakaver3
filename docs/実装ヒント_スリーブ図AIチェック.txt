1. 全体アーキテクチャ（3層構成）
================================================================================

  Layer 1: 幾何学的検出（OpenCV）
    - 通り芯（グリッド線）の座標取得
    - 青いスリーブの位置・サイズ検出
    - 寸法線の検出
    → 出力: ピクセル座標、半径、グリッド位置

  Layer 2: 意味理解（Gemini 2.0 Flash / 2.5 Pro = VLM）
    - スリーブ番号・口径・用途のテキスト読み取り
    - 凡例・注記の解釈
    - 寸法線の起点が何かの意味判定
    → 出力: 構造化メタデータ(JSON)

  Layer 3: 可視化・レポート
    - チェック結果の着色 → OpenCV/Pillowで十分（Nano Banana不要）
    - 最終レポートの見栄え改善 → Nano Banana 2（オプション）
    → 出力: チェック済み図面画像 + Excel/PDFレポート

================================================================================
2. Nano Banana 2 の役割（重要な結論）
================================================================================

  【当初の想定】色を付けるために使う
  【検討結果】座標精度が必要な着色処理にはOpenCVの方が適切

  理由:
  - OpenCV: cv2.circle()等はピクセル単位で正確、決定論的、ミリ秒で完了
  - Nano Banana: 生成AIなので毎回結果が変わる、座標がズレる可能性あり

  【Nano Banana 2 の適切な使い方】
  (1) 最終レポートの見栄え改善（矢印・吹き出しを自然なデザインに）
  (2) デバッグ用途（セマンティックな着色で目視確認の補助）
  (3) 将来的にデモ用の美しいレポート図面生成

  【代わりにGemini VLM（テキストモデル）を活用すべき】
  - 図面を「読んで意味を理解する」能力こそGeminiの真価
  - OpenCVが「どこにスリーブがあるか」→ Geminiが「それが何であるか」
  - この組み合わせが最も強力

  モデルID一覧:
    図面の意味理解（高精度）: gemini-2.5-pro-preview-05-06
    図面の意味理解（軽量）  : gemini-2.0-flash
    画像編集・着色          : gemini-2.0-flash-preview-image-generation
                              ※Nano Banana 2相当

3. OpenCV による通り芯（グリッド線）検出
================================================================================

  手法: モルフォロジー演算（推奨）

  概要:
    1. グレースケール変換 → 適応的二値化
    2. 水平線用カーネル（画像幅の1/3以上）でモルフォロジーオープン
    3. 垂直線用カーネル（画像高さの1/3以上）でモルフォロジーオープン
    4. 射影ヒストグラムでピーク検出 → グリッド線のピクセル座標
    5. 近接ピクセルをクラスタリング（5px以内は同一線）

  コード概念:
    gray = cv2.cvtColor(img, cv2.COLOR_BGR2GRAY)
    binary = cv2.adaptiveThreshold(gray, 255, ...)
    h_kernel = cv2.getStructuringElement(cv2.MORPH_RECT, (w//3, 1))
    h_lines = cv2.morphologyEx(binary, cv2.MORPH_OPEN, h_kernel)
    # → h_linesから各行のピクセル数をカウント → ピーク位置がグリッド線

  ノイズ（寸法線等）との区別:
    - グリッド線は図面の端から端まで走る「長い線」
    - 寸法線は短い → カーネルサイズで自然に除外される
    - 等間隔性の検証: 間隔の標準偏差 < 平均の20% なら等間隔確定
    - 通り芯ラベル（丸囲みのX1,Y1等）をHoughCircles+OCRで紐付け

  通り芯ラベルのOCR:
    - グリッド線の端部（左端/上端）付近で小さな円を検出
    - 円内のテキストをOCR → 通り芯名と座標を紐付け
    - Azure Document Intelligence推奨（.envに設定済み）

4. OpenCV による青いスリーブの検出
================================================================================

  手法: HSV色フィルタ + 輪郭検出（推奨）

  概要:
    1. BGR → HSV変換
    2. 青色範囲でマスク生成: H=100-130, S=80-255, V=80-255
    3. モルフォロジーでノイズ除去（CLOSE→OPEN）
    4. findContoursで個々のスリーブを分離
    5. 各輪郭の中心座標・半径・面積・円形度を算出

  コード概念:
    hsv = cv2.cvtColor(img, cv2.COLOR_BGR2HSV)
    mask = cv2.inRange(hsv, (100,80,80), (130,255,255))
    contours, _ = cv2.findContours(mask, cv2.RETR_EXTERNAL, ...)
    for cnt in contours:
        (cx, cy), radius = cv2.minEnclosingCircle(cnt)
        circularity = 4*pi*area / perimeter^2
        shape = 'circle' if circularity > 0.7 else 'rectangle'

  補完手法（HoughCircles）:
    - 青色マスク上でHoughCirclesを実行 → 円形スリーブを直接検出
    - findContoursの結果と突合して信頼度を上げる

  四角形スリーブ対応:
    - MTGで言及あり（四角いスリーブも存在する）
    - approxPolyDPで頂点数=4 → 四角形と判定
    - minAreaRectで中心・幅・高さ・角度を取得

  精度の見込み:
    - 青色が統一されている場合: 90-98%
    - 注記テキストと被りがある場合: やや低下
    - 解像度300DPI以上推奨

5. スケール変換（ピクセル → 実寸mm）
================================================================================

  手順:
    1. 通り芯間の距離をピクセルで計測（Step 3の結果）
    2. 同じ区間の実寸法をOCRで読み取る（寸法テキスト）
    3. px_per_mm = ピクセル距離 / 実寸法(mm)
    4. 複数区間で算出 → 平均値を採用（一貫性チェック付き）

  精度:
    - 300DPIの場合: 1ピクセル ≒ 0.085mm → ±2-5mmの精度
    - 通り芯間距離が正しく読めれば実用的な精度が出る

  注意:
    - CADスクリーンショットはズーム率で変わる → 必ず図面内寸法を基準
    - PDFの場合はDPI設定が既知なら安定

6. チェック項目の自動化分類
================================================================================

  ランクA（OpenCV + OCR で自動化可能）→ Phase 1-2 で実装
  ────────────────────────────────────────
  #4  通り芯間寸法とスリーブ寄り寸法の合計一致
      → 数値抽出 → 加算 → 照合（算術的に検証可能）
  #9  単一通り芯のみからの寄り寸法でないか
      → 寸法線の参照先が複数通り芯かをOpenCVで検出
  #11 スリーブ芯からの寄り寸法でないか
      → 寸法線の起点がスリーブ中心座標と一致するか幾何判定
  #13 寄り寸法表記の統一性
      → 全スリーブの表記パターンを収集 → 外れ値検出
  #14 スリーブNo記載有無
      → 正規表現 /[A-Z]{1,3}-\d{2,4}/ でマッチ

  ランクB（VLM判定 = Gemini で可能）→ Phase 2 で実装
  ────────────────────────────────────────
  #2  用途・利用設備種別の記載
      → OCR + 設備種別キーワード検索、なければVLMで再確認
  #3  呼び口径・外径の記載
      → /\d+Φ.*外径\d+/ パターン + 呼び径→外径変換テーブル照合
  #8  基準レベルと寸法の記載
      → /\d+FL[+\-]\d+/ パターンマッチ
  #10 型枠段差・凹みからの寄り寸法でないか
      → VLMに寸法線起点のクロップ画像を投入して判定
  #12 柱面・仕上げ面からの寄り寸法でないか
      → 同上、VLMによる意味判定

  ランクC（複数図面照合が必要）→ Phase 3
  ────────────────────────────────────────
  #1  根拠図との整合確認
      → 設備ルート図のスリーブ位置 vs 施工図の位置を座標マッチング
  #5  勾配確保（排水ルート・方向・勾配表記）
      → 排水系統図との照合 + VLM

  ランクD（現時点では自動化困難）→ 手動チェック対象としてリストアップ
  ────────────────────────────────────────
  #6  下階スラブtoスラブの壁・複合耐火壁との干渉
      → 3D的判断が必要
  #7  段差スラブ近辺のスリーブ施工不可場所
      → 構造図との照合 + 3D的判断

8. 入力形式の優先順位
================================================================================

  [最良] DXF/DWG → ezdxf / ODA File Converter
    - 座標・寸法値が正確、レイヤーで要素分離可能
    - 入手できれば最優先

  [推奨] PDF（ベクター） → pdf2image (300DPI)
    - 汎用的、テキスト情報も抽出可能
    - 現実的にはこれが主な入力

  [検証用] CADスクリーンショット / ラスタ画像
    - 解像度・ツールバー混入の問題あり
    - PoCの初期検証には使える

11. OpenCV + VLM 突合のポイント
================================================================================

  OpenCVの強み: 「どこにあるか」（座標が正確）
  Gemini VLMの強み: 「それが何か」（意味が理解できる）

  突合の流れ:
    1. OpenCVで青いスリーブを全数検出 → 座標リスト
    2. 各スリーブ周辺をクロップ（バウンディングボックス + マージン）
    3. クロップ画像をGemini VLMに投入 → テキスト情報抽出
    4. OpenCVの座標 + VLMのメタデータ を結合 → 完全なスリーブ属性テーブル
    5. チェックルールを適用

  ※ OpenCV検出結果の信頼度が低いスリーブはVLMで二重チェック

13. 技術的リスクと対策
================================================================================

  リスク1: 図面の色がバラバラ（現場ごとに異なる）
  → 対策: HSV範囲を設定ファイル化、現場ごとにキャリブレーション

  リスク2: スクリーンショット解像度が低い
  → 対策: PDF入力を推奨、最低300DPI

  リスク3: 寸法線とグリッド線の区別が困難
  → 対策: モルフォロジーの線長フィルタ + 等間隔性検証

  リスク4: VLMのハルシネーション
  → 対策: 信頼度スコア表示 + ヒューマンレビュー必須化

  リスク5: OCRの日本語精度
  → 対策: Azure DI推奨（Tesseractは精度不足の場合が多い）
